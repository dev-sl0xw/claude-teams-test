# AWS 서비스 - Well-Architected Pillar 매핑

## 이 문서의 목적

> **질문: 하나의 AWS 서비스는 하나의 Pillar에만 속할까요?**
>
> 아닙니다. 대부분의 서비스는 여러 Pillar에 걸쳐 있습니다.
> 예를 들어, S3는 보안(암호화), 안정성(내구성 99.999999999%), 비용 최적화(스토리지 클래스)
> 모두에 관련됩니다. 이 매핑표는 각 서비스가 **주로** 어떤 Pillar에서 다뤄지는지 보여줍니다.

## SAA 시험에서 자주 출제되는 서비스 조합

### 고가용성 웹 애플리케이션
```
Route53 → CloudFront → ALB → EC2 (Auto Scaling) → RDS (Multi-AZ)
   │                                                      │
   └─ 안정성 Pillar                            보안 + 안정성 Pillar
```

### 서버리스 아키텍처
```
API Gateway → Lambda → DynamoDB
     │            │         │
     └─ 보안     성능     비용 최적화 + 지속 가능성
```

### 데이터 레이크
```
S3 (수집) → S3 (저장/수명주기) → Athena (분석)
     │              │                  │
    보안          비용 최적화         성능 효율성
```

## 공동 책임 모델 (Shared Responsibility Model)

> **질문: AWS가 책임지는 영역과 고객이 책임지는 영역의 경계는 어디인가요?**

| 구분 | AWS의 책임 | 고객의 책임 |
|------|-----------|-----------|
| 물리적 보안 | ✅ 데이터센터 물리 보안 | |
| 네트워크 인프라 | ✅ 글로벌 네트워크 | ❗ VPC 설정, SG, NACL |
| 컴퓨팅 | ✅ 하이퍼바이저 | ❗ OS 패치, 보안 설정 |
| 스토리지 | ✅ 디스크 관리, 복제 | ❗ 암호화, 접근 제어 |
| 서비스 가용성 | ✅ 서비스 SLA | ❗ Multi-AZ/Multi-Region 설계 |

> **SAA 포인트**: Lambda 같은 관리형 서비스는 고객의 책임 범위가 줄어듭니다.
> 이것이 "서버리스가 보안에도 유리하다"고 말하는 이유입니다.
